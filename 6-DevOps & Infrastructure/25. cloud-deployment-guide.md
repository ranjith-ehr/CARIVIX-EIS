# Cloud Deployment Guide

This document covers the full GCP environment setup, service-by-service deployment procedures, environment variable reference, and rollback instructions.

---

## Prerequisites

Before deploying, ensure the following are in place:

- GCP project `attendance-tracker-prod` exists and billing is enabled.
- The following APIs are enabled:
  - Cloud Run API
  - Cloud SQL Admin API
  - Compute Engine API
  - Cloud Build API
  - Artifact Registry API
  - Secret Manager API
  - Cloud Monitoring API
  - BigQuery API
  - Cloud Memorystore API

Enable all at once:
```bash
gcloud services enable \
  run.googleapis.com \
  sqladmin.googleapis.com \
  compute.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com \
  monitoring.googleapis.com \
  bigquery.googleapis.com \
  redis.googleapis.com \
  --project=attendance-tracker-prod
```

- Terraform 1.6+ installed locally.
- `gcloud` CLI authenticated: `gcloud auth application-default login`
- Docker installed and authenticated to Artifact Registry:
  ```bash
  gcloud auth configure-docker asia-south1-docker.pkg.dev
  ```

## IAM Roles Required

| Service Account | Role | Purpose |
|---|---|---|
| `attendance-api-sa` | `roles/cloudsql.client`, `roles/secretmanager.secretAccessor` | Flask backend Cloud Run service |
| `attendance-pipeline-sa` | `roles/storage.objectAdmin`, `roles/bigquery.dataEditor` | Data pipeline Cloud Run Jobs |
| `attendance-ml-sa` | `roles/storage.objectAdmin`, `roles/bigquery.jobUser` | ML training jobs |
| `attendance-monitoring-sa` | `roles/monitoring.metricWriter`, `roles/logging.logWriter` | Metrics and log export |

Create service accounts:
```bash
for SA in attendance-api-sa attendance-pipeline-sa attendance-ml-sa attendance-monitoring-sa; do
  gcloud iam service-accounts create $SA \
    --display-name="$SA" \
    --project=attendance-tracker-prod
done
```

## Terraform — Infrastructure Provisioning

All infrastructure is defined in `infra/terraform/`. Apply before deploying application code.

```bash
cd infra/terraform/

terraform init -backend-config="bucket=attendance-tfstate-prod"

# Preview changes
terraform plan -var-file="environments/prod.tfvars"

# Apply
terraform apply -var-file="environments/prod.tfvars" -auto-approve
```

### Key Terraform Modules

| Module | Creates |
|---|---|
| `modules/cloud-sql` | PostgreSQL instance, read replica, backup schedule |
| `modules/cloud-run` | Flask backend service, min/max scaling, VPC connector |
| `modules/cloud-run-jobs` | All pipeline and ML training job definitions |
| `modules/memorystore` | Redis instance |
| `modules/gcs-buckets` | All pipeline and artifact storage buckets |
| `modules/gpu-vms` | Face recognition and LLM inference VMs |
| `modules/iam` | Service accounts and role bindings |
| `modules/monitoring` | Alert policies and notification channels |

## Cloud SQL — Deployment

Cloud SQL is provisioned by Terraform. Post-provision steps:

### 1. Run Database Migrations

```bash
# From local with Cloud SQL Auth Proxy running
cloud-sql-proxy attendance-tracker-prod:asia-south1:attendance-db-prod &

export DATABASE_URL="postgresql+psycopg2://admin:${DB_PASS}@127.0.0.1:5432/attendance_db"

cd backend/
alembic upgrade head
```

### 2. Seed Reference Data

```bash
python scripts/seed_db.py --env=production
# Seeds: geofence zones, default attendance rules, admin account
```

### 3. Verify Connection from Cloud Run

```bash
gcloud run services describe attendance-api \
  --region=asia-south1 \
  --format="value(status.url)"
# Hit /health endpoint to confirm DB connectivity
```

## Cloud Run — Flask Backend Deployment

### Build and Push Image

```bash
cd backend/

docker build -t asia-south1-docker.pkg.dev/attendance-tracker-prod/attendance/api:latest .
docker push asia-south1-docker.pkg.dev/attendance-tracker-prod/attendance/api:latest
```

### Deploy to Cloud Run

```bash
gcloud run deploy attendance-api \
  --image=asia-south1-docker.pkg.dev/attendance-tracker-prod/attendance/api:latest \
  --region=asia-south1 \
  --service-account=attendance-api-sa@attendance-tracker-prod.iam.gserviceaccount.com \
  --add-cloudsql-instances=attendance-tracker-prod:asia-south1:attendance-db-prod \
  --set-secrets="DB_URL=db-connection-string:latest,REDIS_URL=redis-connection-string:latest,JWT_SECRET=jwt-secret-key:latest" \
  --min-instances=2 \
  --max-instances=20 \
  --concurrency=80 \
  --cpu=2 \
  --memory=2Gi \
  --timeout=30 \
  --allow-unauthenticated
```

### Verify Deployment

```bash
SERVICE_URL=$(gcloud run services describe attendance-api \
  --region=asia-south1 --format="value(status.url)")

curl -s "${SERVICE_URL}/health" | jq .
# Expected: { "status": "ok", "db": "connected", "redis": "connected" }
```

## GPU Compute Engine VMs — Setup

### Face Recognition VM

```bash
# Create VM
gcloud compute instances create face-recognition-vm \
  --zone=asia-south1-b \
  --machine-type=n1-standard-4 \
  --accelerator=type=nvidia-tesla-t4,count=1 \
  --maintenance-policy=TERMINATE \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --boot-disk-size=50GB \
  --tags=internal-service

# SSH and install dependencies
gcloud compute ssh face-recognition-vm --zone=asia-south1-b

# On the VM:
sudo apt-get install -y python3-pip cuda-toolkit-12-0
pip3 install flask torch torchvision deepface

# Start service (use systemd in production)
cd /app/face-recognition/
python3 server.py --port=5001
```

### LLM Inference VM

```bash
gcloud compute instances create llm-inference-vm \
  --zone=asia-south1-b \
  --machine-type=n1-standard-8 \
  --accelerator=type=nvidia-l4,count=1 \
  --maintenance-policy=TERMINATE \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --boot-disk-size=200GB \
  --tags=internal-service

# On the VM — install vLLM and download model
pip3 install vllm
huggingface-cli download meta-llama/Meta-Llama-3-8B-Instruct --local-dir /models/llama3/

# Start vLLM server
python -m vllm.entrypoints.openai.api_server \
  --model /models/llama3/ \
  --port 8080 \
  --tensor-parallel-size 1 \
  --served-model-name llama3
```

### Systemd Service (Both VMs)

```ini
# /etc/systemd/system/inference-service.service
[Unit]
Description=Inference Service
After=network.target

[Service]
ExecStart=/usr/bin/python3 /app/server.py
Restart=always
RestartSec=5
User=ubuntu

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable inference-service
sudo systemctl start inference-service
```

## Rollback Procedure

### Cloud Run Rollback

List recent revisions:
```bash
gcloud run revisions list \
  --service=attendance-api \
  --region=asia-south1 \
  --sort-by=~CREATED \
  --limit=5
```

Route 100% traffic to a previous revision:
```bash
gcloud run services update-traffic attendance-api \
  --region=asia-south1 \
  --to-revisions=attendance-api-00042-xyz=100
```

### Database Rollback

```bash
# Roll back one migration
alembic downgrade -1

# Roll back to a specific revision
alembic downgrade <revision_id>
```

> Always take a Cloud SQL snapshot before applying migrations in production:
> ```bash
> gcloud sql backups create --instance=attendance-db-prod --project=attendance-tracker-prod
> ```
