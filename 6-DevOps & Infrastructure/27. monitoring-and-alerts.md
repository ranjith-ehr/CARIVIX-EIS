# Monitoring and Alerts

This document covers the metrics collected, Cloud Monitoring dashboard setup, Grafana configuration, alert policy definitions, log-based alerts, and the incident response runbook.

---

## Metrics Collected

All metrics are collected via **Cloud Monitoring** (Google Cloud Operations Suite) with a 60-second scrape interval.

### API Metrics

| Metric | Source | Description |
|---|---|---|
| `custom/api/request_count` | Flask middleware | Total requests per endpoint per minute |
| `custom/api/error_rate` | Flask middleware | Percentage of 5xx responses (1-minute window) |
| `custom/api/latency_p50` | Flask middleware | Median request latency (ms) |
| `custom/api/latency_p99` | Flask middleware | 99th percentile request latency (ms) |
| `custom/api/active_sessions` | Redis | Count of active session tokens in Redis |

### Database Metrics

| Metric | Source | Description |
|---|---|---|
| `cloudsql.googleapis.com/database/cpu/utilization` | Cloud SQL | DB CPU usage |
| `cloudsql.googleapis.com/database/memory/utilization` | Cloud SQL | DB memory usage |
| `cloudsql.googleapis.com/database/postgresql/num_backends` | Cloud SQL | Active DB connections |
| `custom/db/query_latency_p99` | Flask middleware | 99th percentile DB query time (ms) |

### ML Metrics

| Metric | Source | Description |
|---|---|---|
| `custom/ml/inference_latency` | ML service | Time to return a prediction (ms) |
| `custom/ml/anomaly_flag_count` | Pipeline job | Daily anomaly flags generated |
| `custom/ml/false_positive_rate` | Admin Console dismissals | Dismissed alerts / total alerts (7-day rolling) |
| `custom/ml/model_mae` | Scoring job | XGBoost MAE vs actuals (rolling 7-day) |

### Pipeline Metrics

| Metric | Source | Description |
|---|---|---|
| `custom/pipeline/records_processed` | Pipeline job | Records processed per stage per run |
| `custom/pipeline/rejection_rate` | Pipeline job | Rejected records / total records per stage |
| `custom/pipeline/job_duration` | Cloud Run Jobs | Total runtime of each pipeline job |

### Infrastructure Metrics

| Metric | Source | Description |
|---|---|---|
| `run.googleapis.com/request_count` | Cloud Run | Requests per revision |
| `run.googleapis.com/container/cpu/utilization` | Cloud Run | Container CPU utilization |
| `run.googleapis.com/container/memory/utilizations` | Cloud Run | Container memory utilization |
| `compute.googleapis.com/instance/cpu/utilization` | GPU VMs | GPU VM CPU usage |

## Emitting Custom Metrics (Flask Middleware)

```python
# app/middleware/metrics.py
from google.cloud import monitoring_v3
import time

client = monitoring_v3.MetricServiceClient()
project_name = f"projects/{PROJECT_ID}"

def record_request_metric(endpoint: str, status_code: int, latency_ms: float):
    series = monitoring_v3.TimeSeries()
    series.metric.type = "custom.googleapis.com/api/request_count"
    series.metric.labels["endpoint"] = endpoint
    series.metric.labels["status_code"] = str(status_code)

    point = monitoring_v3.Point()
    point.value.int64_value = 1
    now = time.time()
    point.interval.end_time.seconds = int(now)
    series.points = [point]

    client.create_time_series(name=project_name, time_series=[series])
```

The middleware wraps every Flask request using `@app.after_request`.

## Grafana Setup

Grafana is deployed on a separate `e2-medium` VM (`monitoring-vm`) and connects to Cloud Monitoring as a data source.

### Data Source Configuration

1. Open Grafana (`http://monitoring-vm:3000`).
2. Go to **Configuration → Data Sources → Add Data Source**.
3. Select **Google Cloud Monitoring**.
4. Authenticate with the `attendance-monitoring-sa` service account key.
5. Set **Project ID:** `attendance-tracker-prod`.
6. Click **Save & Test**.

### Pre-Built Dashboards

Grafana dashboards are stored as JSON in `infra/grafana/dashboards/`. Import them via:

1. Grafana → **Dashboards → Import**.
2. Upload JSON file or paste content.

| Dashboard File | Description |
|---|---|
| `api-overview.json` | API request rates, latency, error rates |
| `ml-model-health.json` | Inference latency, MAE, false positive rate |
| `pipeline-health.json` | Pipeline job status, rejection rates |
| `infrastructure.json` | Cloud Run, GPU VMs, DB, Redis metrics |

## Alert Policies

All alerts are configured in Cloud Monitoring and notify the **#alerts-prod** Slack channel and the on-call PagerDuty rotation.

### API Alerts

| Alert Name | Condition | Severity | Notification |
|---|---|---|---|
| High Error Rate | Error rate > 5% for 5 consecutive minutes | Critical | PagerDuty + Slack |
| Latency Degradation | P99 latency > 2000ms for 5 minutes | Warning | Slack |
| No Traffic | Request count = 0 for 10 minutes during business hours | Critical | PagerDuty + Slack |

### Database Alerts

| Alert Name | Condition | Severity | Notification |
|---|---|---|---|
| High CPU | DB CPU > 80% for 10 minutes | Warning | Slack |
| Connection Pool Saturation | Active connections > 80% of max_connections | Warning | Slack |
| Query Latency Spike | P99 query time > 500ms for 5 minutes | Warning | Slack |

### ML Alerts

| Alert Name | Condition | Severity | Notification |
|---|---|---|---|
| Model MAE Degradation | XGBoost MAE > 10.0 (7-day rolling) | Warning | ML team Slack |
| High False Positive Rate | Dismissed alerts / total > 20% (7-day) | Warning | ML team Slack |
| Inference Timeout Rate | Inference latency > 8s for > 5% of requests | Warning | Slack |

### Pipeline Alerts

| Alert Name | Condition | Severity | Notification |
|---|---|---|---|
| Pipeline Job Failure | Cloud Run Job exit code != 0 | Critical | PagerDuty + Slack |
| High Rejection Rate | Rejection rate > 5% of records | Warning | Data engineering Slack |
| Pipeline Not Run | Daily pipeline job not completed by 01:00 UTC | Critical | PagerDuty + Slack |

### Infrastructure Alerts

| Alert Name | Condition | Severity | Notification |
|---|---|---|---|
| Cloud Run CPU Throttling | CPU utilization > 90% for 5 minutes | Warning | Slack |
| GPU VM Unreachable | Health check fails for 3 consecutive minutes | Critical | PagerDuty + Slack |
| Redis Memory High | Memory utilization > 80% | Warning | Slack |

## Log-Based Alerts

Log-based alerts trigger on specific log patterns in Cloud Logging:

### Alert: Critical Application Error

**Log filter:**
```
resource.type="cloud_run_revision"
resource.labels.service_name="attendance-api"
severity=CRITICAL
```

**Action:** PagerDuty page + Slack notification.

### Alert: Face Recognition Service Timeout

**Log filter:**
```
resource.type="cloud_run_revision"
jsonPayload.event="face_recognition_timeout"
```

**Action:** Slack notification to #alerts-prod.

### Alert: ML Model Load Failure

**Log filter:**
```
resource.type="cloud_run_job"
jsonPayload.level="ERROR"
jsonPayload.message=~"model load failed"
```

**Action:** Slack notification to ML team channel.

## Incident Response Runbook

### Step 1 — Detect

- Alert fires via PagerDuty or Slack.
- On-call engineer acknowledges within 5 minutes (Critical) or 30 minutes (Warning).

### Step 2 — Triage

1. Open the **Cloud Monitoring Dashboard** to identify the affected service.
2. Check **Cloud Logging** for error messages:
   ```bash
   gcloud logging read \
     'resource.type="cloud_run_revision" AND severity>=ERROR' \
     --limit=50 \
     --project=attendance-tracker-prod \
     --format=json | jq '.[].jsonPayload'
   ```
3. Classify the incident:

| Classification | Description |
|---|---|
| **P1 — Critical** | System unavailable or data loss risk |
| **P2 — High** | Core feature degraded, workaround exists |
| **P3 — Medium** | Non-critical feature affected |
| **P4 — Low** | Minor issue, no user impact |

### Step 3 — Escalate

- P1/P2: Page the secondary on-call and notify #incidents Slack channel.
- P3/P4: Handle in current shift; log in the incident tracker.

### Step 4 — Resolve

Common resolution actions:

| Symptom | Action |
|---|---|
| High API error rate | Check logs for error pattern → fix code → deploy hotfix or rollback |
| DB connection saturation | Scale Cloud Run min instances down, investigate connection leak |
| Pipeline job failure | Check rejection log, identify data issue, reprocess with `--reprocess=true` |
| GPU VM unreachable | SSH to VM, restart systemd service: `sudo systemctl restart inference-service` |
| High ML false positive rate | Trigger manual model retraining from Admin Console |

### Step 5 — Post-Mortem

For P1 and P2 incidents, a post-mortem is required within 48 hours:

- **Template:** Confluence → Engineering → Incidents → Post-Mortem Template
- **Required sections:** Timeline, Root Cause, Impact, Corrective Actions, Action Items with owners and due dates.

## Deployment Annotations

Every production deployment creates a Cloud Monitoring annotation to correlate deployments with metric changes:

```bash
# Added automatically by CI/CD pipeline post-deploy step
gcloud monitoring dashboards describe \
  --project=attendance-tracker-prod \
  <dashboard-id>

# Annotation is visible as a vertical line on all time-series charts
# at the moment of deployment
```
