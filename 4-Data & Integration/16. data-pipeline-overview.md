# Data Pipeline Overview

This document describes the end-to-end data pipeline that transforms raw attendance records into ML-ready feature datasets. It covers each pipeline stage, the tools used, data formats at each transition point, trigger mechanisms, error handling, and data lineage.

---

## Pipeline Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        DATA PIPELINE                            │
│                                                                 │
│  [Raw Ingestion]                                                │
│   Flask Backend → Parquet/CSV (GCS Staging Bucket)             │
│         ↓                                                       │
│  [Data Cleaning]                                                │
│   Pandas → Missing value handling, deduplication,              │
│             timestamp correction, format standardization        │
│         ↓                                                       │
│  [Data Transformation]                                          │
│   Pandas → Work hour calculation, attendance rate,             │
│             late login counts, idle time periods                │
│         ↓                                                       │
│  [Feature Engineering]                                          │
│   Pandas / R → Rolling windows, trend features,                │
│                 deviation scores, streak counts                 │
│         ↓                                                       │
│  [Data Formatting]                                              │
│   Scikit-learn → Min-Max scaling, encoding, schema validation  │
│         ↓                                                       │
│  [ML-Ready Dataset]                                             │
│   Parquet → Consumed by XGBoost, Logistic Regression,          │
│              Isolation Forest, Time-Series models               │
└─────────────────────────────────────────────────────────────────┘
```

## Stage 1 — Raw Ingestion

### Source Systems

| Source | Data Captured | Format |
|---|---|---|
| Flask `attendance_bp` | Login/logout timestamps, face verification result, GIS validation result | PostgreSQL rows |
| Flask `task_bp` | Task descriptions, durations, completion timestamps | PostgreSQL rows |
| Flask `activity_bp` | Session heartbeat events for idle time detection | PostgreSQL rows |

### Ingestion Mechanism

A scheduled Cloud Run Job executes every **15 minutes** and:

1. Queries new records from PostgreSQL since the last ingestion watermark.
2. Serializes records to **Parquet** format (preferred) or CSV fallback.
3. Writes the raw files to the **GCS staging bucket**: `gs://attendance-pipeline-raw/YYYY/MM/DD/`.
4. Updates the ingestion watermark in a control table.

```python
# Ingestion watermark table: pipeline_control
# Columns: job_name (str), last_run_at (timestamp), last_record_id (bigint)
```

### Raw Data Schema (Attendance)

| Column | Type | Description |
|---|---|---|
| `record_id` | UUID | Primary key |
| `employee_id` | UUID | FK to employees table |
| `login_time` | timestamptz | UTC login timestamp |
| `logout_time` | timestamptz | UTC logout timestamp (nullable) |
| `gis_validated` | boolean | GPS zone validation result |
| `face_verified` | boolean | Face recognition result |
| `face_confidence` | float | Match confidence score (0–1) |
| `latitude` | float | Check-in GPS latitude |
| `longitude` | float | Check-in GPS longitude |
| `created_at` | timestamptz | Record creation time |

## Stage 2 — Data Cleaning

**Script:** `pipeline/cleaning/clean_attendance.py`
**Tool:** Pandas

The cleaning stage reads raw Parquet files, applies all validation and correction rules, and outputs a clean Parquet file per day partition.

### Cleaning Operations

| Operation | Logic |
|---|---|
| **Missing logout handling** | Records with `logout_time IS NULL` older than 24 hours are flagged as `incomplete`. Not imputed — left null for downstream exclusion from work hour calculations. |
| **Duplicate removal** | Deduplication key: `(employee_id, DATE(login_time))`. Where duplicates exist, the first valid record (face_verified = true, gis_validated = true) is retained. |
| **Invalid timestamp correction** | Records where `logout_time < login_time` are flagged as `timestamp_error` and excluded from transformation. |
| **Logical consistency check** | `login_time` must precede `logout_time` by at least 1 minute. |
| **Format standardization** | All timestamps converted to UTC. Null `face_confidence` values filled with `0.0`. |
| **Out-of-range GPS** | Latitude must be in [-90, 90], longitude in [-180, 180]. Records outside range flagged as `gps_invalid`. |

### Clean Record Flags

Each cleaned record carries a `data_quality_flag` column:

| Flag | Meaning |
|---|---|
| `clean` | Passed all checks |
| `incomplete` | Missing logout time |
| `timestamp_error` | Logout precedes login |
| `duplicate` | Removed duplicate; original retained |
| `gps_invalid` | GPS coordinates out of valid range |

Only records with flag `clean` are passed to the transformation stage.

### Output

`gs://attendance-pipeline-clean/YYYY/MM/DD/attendance_clean.parquet`

## Stage 3 — Data Transformation

**Script:** `pipeline/transform/transform_attendance.py`
**Tool:** Pandas

Calculates derived metrics from clean records.

### Calculated Metrics

```python
# Work hours per session
df['work_hours'] = (df['logout_time'] - df['login_time']).dt.total_seconds() / 3600

# Attendance rate (per employee, per month)
df['attendance_rate'] = df.groupby(['employee_id', 'year_month'])['status'].transform(
    lambda x: (x.isin(['present', 'late']).sum() / len(x)) * 100
)

# Late login flag
df['is_late'] = df['login_time'].dt.time > late_threshold  # threshold from config

# Idle time (from activity heartbeats join)
df['idle_minutes'] = calculate_idle_time(df['session_id'], heartbeat_df, idle_threshold=30)
```

### Attendance Status Assignment

```python
def assign_status(row):
    if pd.isna(row['login_time']):
        return 'absent'
    elif row['work_hours'] >= FULL_DAY_HOURS:
        return 'late' if row['is_late'] else 'present'
    elif row['work_hours'] >= HALF_DAY_HOURS:
        return 'half_day'
    else:
        return 'absent'
```

### Output

`gs://attendance-pipeline-transformed/YYYY/MM/DD/attendance_transformed.parquet`

## Stage 4 — Feature Engineering

**Script:** `pipeline/features/engineer_features.py`
**Tools:** Pandas, R (optional for statistical features)

Generates behavioral features from transformed records using rolling windows and aggregations.

### Engineered Features

| Feature | Calculation | Window |
|---|---|---|
| `weekly_attendance_avg` | Mean attendance rate | 7-day rolling |
| `monthly_attendance_avg` | Mean attendance rate | 30-day rolling |
| `attendance_consistency_score` | 1 − (std of daily attendance indicator) | 30-day rolling |
| `login_time_deviation` | Std of login hour from employee mean | 30-day rolling |
| `overtime_frequency` | Count of sessions > standard hours / total sessions | 30-day rolling |
| `attendance_streak` | Consecutive present/late days without a break | Running count |
| `work_hour_variance` | Variance of daily work hours | 14-day rolling |
| `late_login_count_30d` | Count of late logins | 30-day rolling |
| `idle_ratio` | Total idle minutes / total session minutes | 30-day rolling |

### Rolling Window Implementation

```python
employee_df = df.sort_values('date').groupby('employee_id')

df['weekly_attendance_avg'] = employee_df['attended'].transform(
    lambda x: x.rolling(window=7, min_periods=1).mean()
)
```

`min_periods=1` ensures features are calculated for employees with fewer than the full window of history.

### Output

`gs://attendance-pipeline-features/YYYY/MM/DD/attendance_features.parquet`

## Stage 5 — Data Formatting

**Script:** `pipeline/format/format_for_ml.py`
**Tool:** Scikit-learn

Prepares the feature dataset for ML model consumption.

### Operations

| Operation | Tool | Notes |
|---|---|---|
| **Min-Max scaling** | `sklearn.preprocessing.MinMaxScaler` | Applied to all continuous numeric features |
| **Categorical encoding** | `sklearn.preprocessing.LabelEncoder` | Applied to `status`, `classification` fields |
| **Schema validation** | Custom validator | Checks column presence, dtypes, null counts |
| **Train/inference split** | Logic flag | `is_training=True` for historical data; `is_inference=True` for latest window |

### Scaler Persistence

The fitted scaler is serialized and stored alongside the model artifacts so that inference-time data uses the same scaling parameters as training data:

```
gs://attendance-ml-artifacts/scalers/minmax_scaler_v{version}.pkl
```

### Output

`gs://attendance-pipeline-ml-ready/YYYY/MM/DD/ml_ready.parquet`

## Trigger Mechanism

| Trigger Type | Frequency | Covers |
|---|---|---|
| **Scheduled ingestion** | Every 15 minutes | New attendance records only |
| **Daily full pipeline run** | 23:30 UTC | Full day's data through all stages |
| **Manual trigger** | On demand (Admin Console or gcloud CLI) | Full backfill or specific date range |

### Manual Trigger (CLI)

```bash
gcloud run jobs execute attendance-pipeline-job \
  --region=asia-south1 \
  --args="--date=2026-04-11,--stage=all"
```

Available `--stage` values: `ingest`, `clean`, `transform`, `features`, `format`, `all`

## Error Handling

| Error Type | Behavior |
|---|---|
| **Stage failure** | The failing stage logs the error, writes a failure record to the `pipeline_run_log` table, and stops. Downstream stages do not run. |
| **Partial record failure** | Individual bad records are written to a rejection log at `gs://attendance-pipeline-rejected/YYYY/MM/DD/`. The stage continues processing valid records. |
| **Retry logic** | Cloud Run Jobs retries failed jobs up to 3 times with exponential backoff (30s, 2m, 8m). |
| **Dead-letter** | After 3 retries, the job failure triggers a Cloud Monitoring alert routed to the data engineering on-call channel. |

### Pipeline Run Log Schema

```sql
CREATE TABLE pipeline_run_log (
    run_id        UUID PRIMARY KEY,
    run_date      DATE,
    stage         VARCHAR(50),
    status        VARCHAR(20),  -- 'success' | 'failed' | 'partial'
    records_in    INTEGER,
    records_out   INTEGER,
    records_rejected INTEGER,
    started_at    TIMESTAMPTZ,
    completed_at  TIMESTAMPTZ,
    error_message TEXT
);
```

## Data Lineage

To trace a specific ML feature back to its raw source:

```
ml_ready.parquet  [feature: login_time_deviation]
    ↑ Calculated in Stage 4 (feature engineering)
    ← attendance_features.parquet [login_time per employee_id]
        ↑ Derived in Stage 3 (transformation)
        ← attendance_transformed.parquet [login_time, is_late]
            ↑ Cleaned in Stage 2
            ← attendance_clean.parquet [login_time, data_quality_flag = 'clean']
                ↑ Ingested in Stage 1
                ← PostgreSQL attendance_logs [record_id, login_time]
```

Each stage output file includes a `pipeline_run_id` column linking back to the `pipeline_run_log` table for full audit traceability.
