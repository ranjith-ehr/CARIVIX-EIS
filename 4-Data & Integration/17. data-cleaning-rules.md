# Data Cleaning Rules

This document specifies every cleaning rule applied in Stage 2 of the data pipeline. Each rule includes its detection logic, the action taken, the output flag assigned, and the downstream impact of exclusion or correction.

---

## Overview

Cleaning is applied to three datasets ingested from PostgreSQL:

| Dataset | Source Table | Pipeline Script |
|---|---|---|
| Attendance records | `attendance_logs` | `pipeline/cleaning/clean_attendance.py` |
| Task logs | `task_logs` | `pipeline/cleaning/clean_tasks.py` |
| Activity heartbeats | `activity_events` | `pipeline/cleaning/clean_activity.py` |

All three produce a cleaned Parquet output and a **rejection log** for records that cannot be corrected.

## Attendance Records — Cleaning Rules

### Rule A1 — Missing Logout Time

**Detection:**
```python
df['logout_time'].isna() & (df['login_time'] < now - timedelta(hours=24))
```

**Action:** Flag as `incomplete`. Do not impute. Exclude from work hour calculations and attendance status assignment (treated as absent if no manual correction exists).

**Output flag:** `incomplete`

**Downstream impact:** Record is retained in the clean dataset with null `work_hours`. Excluded from performance score calculation for that session. Visible in the Manager Dashboard as an incomplete session.

**Note:** Records with a missing logout where `login_time` is within the last 24 hours are treated as **in-progress sessions** and are not flagged — they are not included in the daily pipeline run until the session window closes.

---

### Rule A2 — Duplicate Records

**Detection:**
```python
duplicates = df.duplicated(subset=['employee_id', df['login_time'].dt.date], keep=False)
```

**Action:** Among duplicates, retain the record with the highest quality score:

```python
def quality_priority(row):
    score = 0
    if row['face_verified']: score += 2
    if row['gis_validated']: score += 2
    if not pd.isna(row['logout_time']): score += 1
    return score
```

The highest-scoring record is kept; others are written to the rejection log with flag `duplicate`.

**Output flag:** `duplicate` (on rejected copies); `clean` (on retained record after passing other checks)

**Downstream impact:** Only one attendance record per employee per day enters downstream stages. Deduplication prevents inflated attendance counts.

---

### Rule A3 — Invalid Timestamp Order

**Detection:**
```python
df['logout_time'] < df['login_time']
```

**Action:** Flag as `timestamp_error`. Exclude from transformation. Written to rejection log.

**Output flag:** `timestamp_error`

**Downstream impact:** Record is excluded entirely. Does not contribute to work hours, attendance rate, or performance score. An HR admin must manually correct and re-submit the record.

---

### Rule A4 — Zero-Duration Sessions

**Detection:**
```python
(df['logout_time'] - df['login_time']).dt.total_seconds() < 60
```

**Action:** Flag as `zero_duration`. Exclude from work hour calculations. Treated as a failed or accidental check-in/out pair.

**Output flag:** `zero_duration`

**Downstream impact:** Excluded from work hours and performance score. Appears in the anomaly detection model input as a potential suspicious login pattern signal.

---

### Rule A5 — Out-of-Range GPS Coordinates

**Detection:**
```python
~df['latitude'].between(-90, 90) | ~df['longitude'].between(-180, 180)
```

**Action:** Flag as `gps_invalid`. GPS fields set to null. Record retained for attendance status purposes but GIS validation result is overridden to `False`.

**Output flag:** `gps_invalid`

**Downstream impact:** Attendance status can still be assigned (based on timestamp only). GIS validation is recorded as failed. The record may trigger a location anomaly alert.

---

### Rule A6 — Missing Face Confidence Score

**Detection:**
```python
df['face_verified'] == True & df['face_confidence'].isna()
```

**Action:** Impute `face_confidence` with `0.5` (neutral value). Flag the imputation in a separate `face_confidence_imputed` boolean column.

**Output flag:** `clean` (with `face_confidence_imputed = True`)

**Downstream impact:** Minimal — confidence score is not a primary feature in ML models but is used as a weak signal in anomaly scoring.

---

### Rule A7 — Future-Dated Records

**Detection:**
```python
df['login_time'] > pd.Timestamp.utcnow()
```

**Action:** Flag as `future_dated`. Exclude entirely. Written to rejection log.

**Output flag:** `future_dated`

**Downstream impact:** Excluded from all calculations. Likely a system clock error on the client device; DevOps should be notified if volume of future-dated records exceeds 5/day.

---

### Rule A8 — Timezone Normalization

**Detection:** All records on ingestion.

**Action:** Convert all `login_time` and `logout_time` values to UTC using `pd.Timestamp.tz_convert('UTC')`. Source timezone is derived from the employee's configured timezone in the `employees` table.

**Output flag:** No flag — applied universally.

**Downstream impact:** Ensures consistent timestamp arithmetic across employees in different timezones.

---

## Task Logs — Cleaning Rules

### Rule T1 — Missing Task Description

**Detection:**
```python
df['description'].isna() | (df['description'].str.strip() == '')
```

**Action:** Flag as `missing_description`. Retain record for duration-based features but exclude from NLP/chatbot summarization.

**Output flag:** `missing_description`

---

### Rule T2 — Negative or Zero Duration

**Detection:**
```python
df['duration_minutes'] <= 0
```

**Action:** Flag as `invalid_duration`. Exclude from feature engineering.

**Output flag:** `invalid_duration`

---

### Rule T3 — Duration Outlier

**Detection:**
```python
df['duration_minutes'] > 720  # more than 12 hours on a single task
```

**Action:** Flag as `duration_outlier`. Retain but cap at 720 minutes for feature engineering to prevent outlier distortion.

**Output flag:** `duration_outlier`

---

### Rule T4 — Orphaned Task Records

**Detection:** Task records with `employee_id` not present in the `employees` table (FK violation in the source).

**Action:** Flag as `orphaned`. Write to rejection log. Do not include in any downstream processing.

**Output flag:** `orphaned`

---

## Activity Heartbeats — Cleaning Rules

### Rule H1 — Out-of-Session Heartbeats

**Detection:** Heartbeat `timestamp` outside the `[login_time, logout_time]` window for the linked session.

**Action:** Discard. These are likely caused by clock drift or late-arriving events.

---

### Rule H2 — Duplicate Heartbeats

**Detection:** Same `(session_id, timestamp)` pair appearing more than once.

**Action:** Deduplicate; keep one. No downstream impact.

---

### Rule H3 — Heartbeat Gap Calculation

Not a cleaning rule per se, but the idle time calculation depends on correct heartbeat ordering:

```python
df_sorted = df.sort_values(['session_id', 'timestamp'])
df_sorted['gap_minutes'] = df_sorted.groupby('session_id')['timestamp'].diff().dt.total_seconds() / 60
idle_periods = df_sorted[df_sorted['gap_minutes'] > idle_threshold]  # default: 30 minutes
```

## Rejection Log Schema

All rejected records are written to:
`gs://attendance-pipeline-rejected/YYYY/MM/DD/{dataset}_rejected.parquet`

```python
rejection_log_schema = {
    'source_record_id': 'UUID',
    'dataset':          'str',       # 'attendance' | 'tasks' | 'activity'
    'rule_code':        'str',       # e.g., 'A3', 'T1'
    'flag':             'str',       # e.g., 'timestamp_error'
    'original_record':  'json',      # full original record as JSON string
    'rejected_at':      'timestamp',
    'pipeline_run_id':  'UUID'
}
```

Rejection logs are retained for **90 days** and are queryable for audit or reprocessing purposes.

## Reprocessing Rejected Records

After a manual correction (e.g., HR fixes a timestamp error via the Admin Console), the record can be reprocessed:

```bash
gcloud run jobs execute attendance-pipeline-job \
  --region=asia-south1 \
  --args="--date=2026-04-11,--stage=clean,--reprocess=true"
```

The `--reprocess` flag re-runs cleaning for the specified date partition, replacing the existing clean output. Transformation and subsequent stages must also be re-run to propagate the correction.

---

## Cleaning Quality Thresholds

The pipeline raises an alert if the rejection rate for any stage exceeds the following thresholds:

| Metric | Alert Threshold |
|---|---|
| Attendance records rejected | > 5% of daily records |
| Task records rejected | > 10% of daily records |
| Future-dated records | > 5 records/day |
| Timestamp errors | > 2% of daily records |

Alerts are sent to the data engineering on-call channel via Cloud Monitoring.
