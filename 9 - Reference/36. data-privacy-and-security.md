# Data Privacy and Security

This document describes what data the Attendance Tracking system collects, how it is stored and protected, how long it is retained, and what rights employees have over their data.

---

## Data Collected Per Employee

The following data is collected and stored for each employee account:

| Data Type | What Is Captured | When |
|---|---|---|
| **Attendance Timestamps** | Login time, logout time | At each check-in and check-out |
| **Attendance Status** | Present, Late, Half-Day, Absent | Computed daily |
| **GPS Coordinates** | Latitude and longitude at check-in | At each check-in |
| **GIS Validation Result** | Pass / Fail, zone name | At each check-in |
| **Face Verification Result** | Pass / Fail, confidence score | At each check-in and check-out |
| **Facial Embeddings** | Numerical vector (not a photo) | At enrollment; updated on re-enrollment |
| **Task Logs** | Task description, duration, timestamp | When logged by the employee |
| **Activity Heartbeats** | Session activity events for idle time detection | During active sessions |
| **Performance Scores** | Composite score and component values | Nightly calculation |
| **Anomaly Flags** | Anomaly type, severity, date | When flagged by the detection model |

### What Is NOT Stored

- Raw photographs or video from face verification sessions.
- Audio recordings of any kind.
- Browser history, keystrokes, or screen content.
- Device identifiers beyond what is required for session management.

---

## Facial Embedding Storage

Facial embeddings receive additional protection given their sensitivity as biometric data.

| Detail | Policy |
|---|---|
| **Format** | Numerical vector — not a photograph and not human-readable |
| **Storage location** | Encrypted column in PostgreSQL (Cloud SQL) |
| **Encryption** | AES-256 encryption at rest |
| **Access control** | Accessible only by the ML inference service; no human account has direct read access |
| **Transmission** | Transmitted over TLS 1.3 between device and server; never logged in plaintext |
| **Retention** | Deleted within 30 days of account deactivation |
| **Raw images** | Frames captured during verification are processed in memory and not written to disk or any storage layer |

---

## Authentication and Session Security

| Mechanism | Detail |
|---|---|
| **Authentication** | JWT (JSON Web Token), signed with HS256 |
| **Token expiry** | 8 hours |
| **Token refresh** | Supported via `/auth/refresh`; requires a valid non-expired token |
| **Token revocation** | Tokens are invalidated on logout; revoked tokens cannot be refreshed |
| **Password storage** | Passwords are hashed using bcrypt (cost factor 12); plaintext passwords are never stored |
| **Failed login policy** | Account locked after 5 consecutive failed attempts; requires admin unlock |
| **Transport security** | All API communication over HTTPS (TLS 1.3); HTTP requests are redirected to HTTPS |

## Role-Based Access Control

Access to data is strictly governed by role. The table below summarizes read and write permissions per data type.

| Data Type | Employee | Manager | Admin |
|---|---|---|---|
| Own attendance records | Read | Read | Read / Write |
| Team attendance records | ❌ | Read | Read / Write |
| Own performance score | Read | Read | Read |
| Team performance scores | ❌ | Read | Read |
| Own anomaly flags | Read (view only) | Read / Write | Read / Write |
| Team anomaly flags | ❌ | Read / Write | Read / Write |
| User account data | Own profile only | Own profile only | Read / Write (all) |
| Facial embeddings | ❌ | ❌ | ❌ (ML service only) |
| Audit logs | ❌ | ❌ | Read |
| System configuration | ❌ | ❌ | Read / Write |

---

## Data Retention Policy

| Data Type | Retention Period | Action After Retention |
|---|---|---|
| Attendance records | 12 months | Archived; accessible to Admin on request |
| Performance scores | 12 months | Archived |
| Task logs | 12 months | Archived |
| Anomaly flags | 12 months | Deleted |
| Facial embeddings | Until 30 days after account deactivation | Permanently deleted |
| Audit logs | 12 months | Deleted |
| Activity heartbeats | 90 days | Deleted |
| GPS coordinates | 12 months | Archived |

Default retention periods can be extended for compliance purposes by your DevOps team.

---

## Data Subject Rights

Employees have the following rights over their personal data:

| Right | How to Exercise |
|---|---|
| **Access** | View your attendance records, performance scores, and task logs directly in the app under **My Attendance** and **Performance**. |
| **Correction** | Request corrections to inaccurate attendance records through your manager or HR team. |
| **Deletion** | Request account deletion through your HR team. Upon deactivation, personal data is retained per the retention schedule above, then deleted. Facial embeddings are deleted within 30 days. |
| **Portability** | Export your attendance records as CSV from **My Attendance → Export**. |
| **Objection** | Contact your HR team if you have concerns about how specific data is being used. |

## Compliance Notes

The system is designed to align with the following regulatory frameworks. Organizations deploying this system are responsible for ensuring their specific configuration and use of the platform meets applicable local regulations.

| Framework | Relevant Aspects |
|---|---|
| **GDPR (EU)** | Lawful basis for processing biometric data; data minimization; right to erasure; data retention limits |
| **PDPA (India)** | Consent for biometric data collection; data localization considerations |
| **ISO 27001** | Access control, audit logging, encryption at rest and in transit, incident management |

> **Note:** This document describes the system's built-in data handling capabilities. Your organization's specific compliance obligations depend on your jurisdiction, industry, and internal policies. Consult your legal or compliance team for guidance.

## Security Incident Reporting

If you suspect unauthorized access to your account or believe your credentials have been compromised:

1. Immediately change your password via **Profile → Settings → Change Password**.
2. Log out of all active sessions via **Profile → Sign Out All Devices**.
3. Notify your system administrator.

Administrators should report suspected security incidents to the DevOps team, who will initiate the incident response runbook.
