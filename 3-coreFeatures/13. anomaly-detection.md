# Anomaly Detection

The anomaly detection system automatically identifies attendance behaviors that fall outside an employee's normal patterns. It runs continuously in the background and routes alerts to the relevant manager for review.

---

## What Counts as an Anomaly

An anomaly is any attendance event that deviates significantly from an employee's established behavioral baseline. The system does not apply fixed global thresholds — it learns each employee's individual patterns and flags deviations relative to that baseline.

### Anomaly Types

| Type | Description | Example |
|---|---|---|
| **Sudden Absenteeism** | Unexpected consecutive absences after a consistent attendance history | Employee with 95% attendance rate misses 4 days without prior leave request |
| **Suspicious Login Pattern** | Login times that deviate sharply from the employee's established routine | Employee who consistently logs in at 08:55 suddenly logs in at 06:10 for multiple days |
| **Unrealistic Work Hours** | Sessions with abnormally short or excessively long durations | 14-minute session recorded as a full day; 22-hour session with no break |
| **Location Mismatch** | Check-in from outside the approved geofence zone | Employee checks in from a location 3 km from the office |
| **Simultaneous Location** | Same account appearing in two locations within an impossible travel window | Check-in from two cities 500 km apart within 30 minutes |
| **Rapid Consecutive Check-Ins** | Multiple check-in attempts within a very short window | 5 check-in attempts within 2 minutes |

---

## How the Model Works

The system uses the **Isolation Forest** algorithm — an unsupervised machine learning model that assigns an anomaly score to each attendance record.

In plain terms:

- The model is trained on the full history of employee attendance data.
- Records that are easily "isolated" from the majority of normal data points receive a high anomaly score.
- Records above a configured score threshold are flagged as anomalies.

The model runs as a scheduled batch job after each day's attendance data is processed. It does not evaluate records in real time during check-in (GIS validation handles real-time location checks).

## Anomaly Lifecycle

```
Record flagged by Isolation Forest
          ↓
Alert created → routed to employee's manager
          ↓
Manager reviews alert detail
          ↓
    ┌─────────────────────────────────┐
    │  Action:                        │
    │  Acknowledge → No further steps │
    │  Escalate → Routed to HR/Admin  │
    │  Dismiss → False positive log   │
    └─────────────────────────────────┘
          ↓
Alert status updated; logged in Audit Log
```

---

## Alert Severity Levels

| Severity | Description | Examples |
|---|---|---|
| **Low** | Minor deviation; likely explainable | Slightly unusual login time on a single day |
| **Medium** | Moderate deviation; warrants review | Repeated late logins outside normal pattern; single location mismatch |
| **High** | Significant deviation; requires prompt review | Simultaneous location detection; sudden multi-day absence; unrealistic work hours |

---

## Alert Routing

- Alerts are routed to the **direct manager** of the flagged employee.
- High-severity alerts also trigger an email notification to the manager.
- If no manager is assigned, alerts are routed to the **Admin**.
- Managers can escalate any alert to HR or Admin via the alert detail panel.

---

## Reviewing and Resolving Alerts

Managers access alerts via **Alerts → Anomaly Flags** or the dashboard alert count badge.

### Alert Detail View
Each alert shows:
- Employee name and department
- Anomaly type and severity
- Date and time of the flagged event
- Flagged session data (login time, logout time, location, work hours)
- Anomaly score (0–1 scale; higher = more anomalous)

### Resolution Actions

| Action | When to Use | Requires Note? |
|---|---|---|
| **Acknowledge** | Reviewed; no action needed | Optional |
| **Escalate** | Needs HR or Admin involvement | Required |
| **Dismiss as False Positive** | System error; behavior was legitimate | Required |

False positive dismissals are collected and used to improve model accuracy during the next retraining cycle.

---

## Employee Visibility

Employees can see whether their account has active anomaly flags under **My Attendance → Flags**. They can view the anomaly type and date but cannot resolve or dismiss alerts — that action is reserved for managers and admins.
