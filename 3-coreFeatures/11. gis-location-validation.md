# GIS Location Validation

GIS (Geographic Information System) location validation confirms that an employee is physically present within an approved office zone at the time of check-in. This prevents remote or unauthorized check-ins and detects location-based attendance anomalies.

---

## How Geofencing Works

A **geofence** is a virtual boundary drawn around a physical location — for example, an office building or campus. Each geofence zone is defined by an administrator as either:

- **Radius-based** — A circular boundary with a centre point (latitude, longitude) and a radius in meters.
- **Polygon-based** — A custom-shaped boundary with multiple vertices, used for non-circular office layouts.

During check-in, the employee's device GPS coordinates are compared against all active geofence zones assigned to them. If the coordinates fall within at least one zone, the location check passes.

---

## Check-In Validation Flow

```
Employee taps Check In
        ↓
GPS coordinates captured from device
        ↓
Coordinates compared against assigned geofence zones
        ↓
   ┌────────────────────────────────┐
   │  Within zone?                  │
   │  Yes → Location Verified ✅    │
   │  No  → Location Alert ❌       │
   └────────────────────────────────┘
        ↓
Additional checks run (travel plausibility, duplicate location)
        ↓
Result logged with check-in record
```

---

## Validation Checks Performed

Beyond basic zone membership, the system runs the following checks on every session:

### 1. Zone Membership
Verifies that the check-in coordinates fall within at least one assigned geofence zone.

### 2. Travel Distance Plausibility
Checks whether the distance between the employee's current location and their previous check-in location is realistic given the time elapsed.

**Example:** If an employee checked in from Mumbai at 09:00 and attempts to check in from Delhi at 09:15, the system flags this as a plausibility violation — the distance cannot be covered in 15 minutes.

### 3. Simultaneous Location Detection
Detects if the same employee account appears to be checking in from two different locations at the same time, or within an impossible travel window.

---

## Alert Types

| Alert Type | Trigger | Severity |
|---|---|---|
| **Outside Zone** | Check-in coordinates fall outside all assigned geofence zones | Medium |
| **Unrealistic Travel** | Implausible distance covered in the time since last check-in | High |
| **Simultaneous Location** | Same account, two locations within an impossible travel window | High |
| **Repeated Zone Failure** | 3 or more failed location checks within a 7-day period | Medium |

Alerts are visible to the employee's manager and routed to the anomaly review queue. See [Anomaly Detection](./anomaly-detection.md) for how alerts are reviewed and resolved.

---

## What Happens When Location Validation Fails

If the check-in location does not pass validation:

1. The employee sees a **Location Not Verified** message on screen.
2. The check-in is **not recorded**.
3. The employee may retry after moving to a verified location.
4. If the failure persists while the employee is physically present, they should contact their manager to log a manual attendance record.
5. A location alert is generated and routed to the manager's anomaly queue.

## Admin Configuration

Administrators configure geofence zones under **Admin Console → Geofences**. See the [Admin User Guide](../user-guides/admin-user-guide.md) for setup steps.

Key configuration parameters:

| Parameter | Description |
|---|---|
| **Zone Name** | Descriptive label (e.g., "HQ Building A") |
| **Boundary Type** | Radius or Polygon |
| **Centre / Radius** | For radius zones: GPS coordinates + meters |
| **Vertices** | For polygon zones: ordered list of GPS coordinates |
| **Assignment** | Assigned to specific departments or all employees |
| **Buffer Tolerance** | Additional meters added to the boundary to account for GPS drift (default: 20 m) |

---

## Accuracy Limitations and Edge Cases

| Scenario | Behavior |
|---|---|
| **Indoor GPS drift** | GPS accuracy degrades indoors. A 20-meter buffer tolerance is applied by default to account for this. |
| **High-rise buildings** | GPS cannot determine floor level. Zone boundaries apply horizontally only. |
| **VPN usage** | Location is validated from device GPS, not IP address. VPN does not affect the result. |
| **Airplane/Low signal mode** | GPS will not lock. Check-in cannot proceed until signal is restored. |
| **GPS spoofing attempts** | The system cross-references GPS data with device sensor data. Spoofing attempts are flagged as high-severity anomalies. |
