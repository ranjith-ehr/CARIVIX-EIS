# Model Optimization

This document covers hyperparameter tuning procedures, SHAP value generation and interpretation, imbalanced dataset handling, cross-validation strategy, and the full model retraining and deployment workflow.

---

## Hyperparameter Tuning

### XGBoost — Performance Predictor

Tuning is performed using **Bayesian Optimization** via `scikit-optimize` (`BayesSearchCV`). GridSearchCV was evaluated but Bayesian search converged on equivalent results in ~60% fewer evaluations.

```python
from skopt import BayesSearchCV
from xgboost import XGBRegressor

search_space = {
    'n_estimators':     (100, 1000),
    'max_depth':        (3, 10),
    'learning_rate':    (0.01, 0.3, 'log-uniform'),
    'subsample':        (0.6, 1.0),
    'colsample_bytree': (0.6, 1.0),
    'reg_alpha':        (0.0, 1.0),
    'reg_lambda':       (0.5, 5.0)
}

opt = BayesSearchCV(
    XGBRegressor(random_state=42, tree_method='hist'),
    search_space,
    n_iter=50,
    cv=5,
    scoring='neg_mean_absolute_error',
    n_jobs=-1,
    random_state=42
)
opt.fit(X_train, y_train)
best_params = opt.best_params_
```

**Current best parameters (xgb_v3):**

| Parameter | Value |
|---|---|
| `n_estimators` | 650 |
| `max_depth` | 6 |
| `learning_rate` | 0.042 |
| `subsample` | 0.82 |
| `colsample_bytree` | 0.74 |
| `reg_alpha` | 0.18 |
| `reg_lambda` | 2.1 |

### Logistic Regression — Employee Classifier

Tuned using `GridSearchCV` over the regularization parameter `C` and the solver:

```python
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import GridSearchCV

param_grid = {
    'C': [0.01, 0.05, 0.1, 0.5, 1.0, 5.0, 10.0],
    'solver': ['lbfgs', 'saga'],
    'max_iter': [1000]
}

grid = GridSearchCV(
    LogisticRegression(class_weight='balanced', random_state=42),
    param_grid,
    cv=5,
    scoring='f1_weighted',
    n_jobs=-1
)
grid.fit(X_train, y_train)
```

**Current best parameters (lr_v2):** `C=0.5`, `solver='lbfgs'`

### Isolation Forest — Anomaly Detector

Isolation Forest has limited hyperparameters to tune. The primary control is `contamination`. Tuning is done manually via threshold sweep:

```python
contamination_values = [0.02, 0.03, 0.05, 0.07, 0.10]

for c in contamination_values:
    model = IsolationForest(n_estimators=200, contamination=c, random_state=42)
    model.fit(X_train)
    scores = model.decision_function(X_val)
    # Evaluate against labeled anomaly test set
    precision, recall, _ = precision_recall_curve(y_val_labels, -scores)
    print(f"contamination={c}: AUC-PR={auc(recall, precision):.3f}")
```

Current setting: `contamination=0.05` (AUC-PR = 0.81 on labeled test set).

### Prophet — Attendance Forecaster

Prophet's key tunable parameter is `changepoint_prior_scale`, which controls flexibility in trend changes. Tuned via walk-forward cross-validation:

```python
from prophet.diagnostics import cross_validation, performance_metrics

for scale in [0.01, 0.05, 0.10, 0.50]:
    m = Prophet(changepoint_prior_scale=scale, yearly_seasonality=True, weekly_seasonality=True)
    m.fit(df_train)
    df_cv = cross_validation(m, initial='180 days', period='30 days', horizon='7 days')
    df_perf = performance_metrics(df_cv)
    print(f"scale={scale}: MAPE={df_perf['mape'].mean():.4f}")
```

Current setting: `changepoint_prior_scale=0.05` (MAPE = 0.072).

## Imbalanced Dataset Handling

### Logistic Regression (Employee Classifier)

High-risk employees are a minority class (~10–15% of employees). Two strategies are applied:

**Strategy 1 — Class Weighting (primary)**
```python
LogisticRegression(class_weight='balanced')
# Scikit-learn automatically sets weights inversely proportional to class frequency
```

**Strategy 2 — SMOTE (backup, used when high_risk < 8% of training data)**
```python
from imblearn.over_sampling import SMOTE

sm = SMOTE(random_state=42, k_neighbors=5)
X_resampled, y_resampled = sm.fit_resample(X_train, y_train)
```

SMOTE is applied before fitting, not inside the CV loop. The SMOTE-resampled dataset is used only for training; evaluation always uses the original unbalanced test set.

### Isolation Forest (Anomaly Detector)

Isolation Forest is inherently robust to imbalanced data as an unsupervised model. No oversampling is applied. The `contamination` parameter implicitly handles the expected class ratio.

## Cross-Validation Strategy

### Standard Models (XGBoost, Logistic Regression)

5-fold stratified cross-validation is used for all hyperparameter tuning:

```python
from sklearn.model_selection import StratifiedKFold, TimeSeriesSplit

# For classification (Logistic Regression)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)

# For regression (XGBoost) — time-aware split to prevent data leakage
cv = TimeSeriesSplit(n_splits=5)
```

`TimeSeriesSplit` is used for XGBoost to prevent future data from leaking into training folds — a critical issue for time-dependent attendance features like rolling windows.

### Attendance Forecaster (Prophet)

Walk-forward validation is used instead of k-fold:

```python
df_cv = cross_validation(
    model,
    initial='180 days',   # minimum training data
    period='30 days',     # step between cutoff dates
    horizon='7 days'      # forecast window to evaluate
)
```

## SHAP Value Generation

SHAP (SHapley Additive exPlanations) is used to explain individual performance score predictions and generate the plain-language score breakdown shown on the employee dashboard.

### Generating SHAP Values

```python
import shap

explainer = shap.TreeExplainer(xgb_model)
shap_values = explainer.shap_values(X_employee)

# shap_values[i] = contribution of feature i to the prediction
# Positive = pushes score up; Negative = pushes score down
```

### Summary Plot (Model-Level Explainability)

```python
shap.summary_plot(shap_values, X_train, feature_names=feature_cols)
```

Generates a beeswarm plot showing which features have the greatest impact across all employees. Used by the ML team to validate feature importance and detect drift.

### Force Plot (Individual Prediction)

```python
shap.force_plot(
    explainer.expected_value,
    shap_values[employee_idx],
    X_employee.iloc[employee_idx],
    feature_names=feature_cols
)
```

### Plain-Language Summary Generation

The score breakdown text shown to employees is generated from SHAP values:

```python
def generate_shap_summary(shap_values, feature_names, feature_values):
    contributions = sorted(
        zip(feature_names, shap_values, feature_values),
        key=lambda x: abs(x[1]),
        reverse=True
    )
    top_positive = [(f, v) for f, v, _ in contributions if v > 0][:2]
    top_negative = [(f, v) for f, v, _ in contributions if v < 0][:2]

    lines = []
    for feat, val in top_positive:
        lines.append(f"{feature_display_name(feat)} contributed positively (+{val:.1f} pts).")
    for feat, val in top_negative:
        lines.append(f"{feature_display_name(feat)} reduced your score ({val:.1f} pts).")
    return ' '.join(lines)
```

The `feature_display_name()` mapping is defined in `ml/config/feature_display_names.yaml`.

## Model Retraining Workflow

### Trigger Conditions

| Trigger | Details |
|---|---|
| Scheduled retraining | See retraining schedule in `ml/config/schedule.yaml` |
| Manual trigger (Admin Console) | Queues a Cloud Run Job immediately |
| Drift detection | Automated alert if model performance degrades below threshold over 7-day monitoring window |
| Data event | Triggered automatically when employee count changes by > 10% |

### Retraining Steps

```
1. Load ML-ready dataset (last N months from GCS)
        ↓
2. Split: train (80%) / test (20%) — time-ordered
        ↓
3. Hyperparameter tuning (BayesSearchCV / GridSearchCV)
        ↓
4. Train model with best params on full training set
        ↓
5. Evaluate on test set against metric thresholds
        ↓
6. If thresholds met:
       Serialize model + scaler to GCS with new version number
       Update model registry (metadata.json)
       Promote to production (update serving config)
   If thresholds not met:
       Log failure, retain current production model
       Alert ML team Slack channel
        ↓
7. Run ML test suite (see ml-model-testing.md)
        ↓
8. Update `performance_scores` table with new model_version field
```

### Retraining Script Entry Point

```bash
python ml/train/run_training.py \
  --model=xgboost \
  --data-from=2025-04-01 \
  --data-to=2026-03-31 \
  --output-bucket=gs://attendance-ml-artifacts/performance-predictor/
```

Available `--model` values: `xgboost`, `logistic`, `isolation_forest`, `prophet`

## Model Monitoring and Drift Detection

Post-deployment, the following metrics are tracked daily via Cloud Monitoring:

| Metric | Alert Threshold |
|---|---|
| XGBoost MAE (on recent actuals vs predictions) | > 10.0 |
| Classifier high_risk precision (rolling 7 days) | < 0.65 |
| Anomaly false positive rate (dismissed/total) | > 20% |
| Forecast MAPE | > 15% |

When a threshold is breached, an alert is sent to the ML on-call channel and a manual retraining review is initiated.
