# NLP Chatbot Architecture

This document describes the full architecture of the AI chatbot — from user query to response — covering intent detection, named entity recognition, retrieval augmented generation, LLM inference, and fallback handling.

---

## Component Overview

```
User Query (natural language)
         ↓
┌─────────────────────────────────┐
│  1. Intent Detection            │
│     DistilBERT / RoBERTa        │
│     → intent label + confidence │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  2. Named Entity Recognition    │
│     spaCy                       │
│     → DATE, EMPLOYEE, METRIC    │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  3. Data Retrieval              │
│     PostgreSQL / Redis query    │
│     → structured data records   │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  4. RAG Pipeline                │
│     LangChain + FAISS/Pinecone  │
│     → retrieved context chunks  │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  5. LLM Response Generation     │
│     LLaMA 3 / Mistral           │
│     → natural language response │
└─────────────────────────────────┘
```

## Component 1 — Intent Detection

### Model

**Primary:** `distilbert-base-uncased` fine-tuned on the attendance query corpus  
**Fallback:** `roberta-base` (used when DistilBERT confidence < 0.6)

### Intent Taxonomy

| Intent Label | Description | Example Query |
|---|---|---|
| `attendance_query` | Attendance rate, status, absent/present days | "What is my attendance rate this month?" |
| `work_hours_query` | Work hours, session duration, overtime | "Show my work hours for last week." |
| `late_login_query` | Late login counts or specific late events | "How many times was I late in March?" |
| `performance_query` | Performance score, score breakdown, trend | "Why did my score drop this week?" |
| `task_query` | Task logs, task count, task summary | "Summarize my tasks from yesterday." |
| `team_attendance_query` | Manager team-level attendance | "Who was absent today?" |
| `anomaly_query` | Anomaly flags and alerts | "Are there any unresolved anomaly flags?" |
| `forecast_query` | Attendance or performance predictions | "What is the predicted attendance next Monday?" |
| `classification_query` | Employee classification (manager only) | "List high-risk employees." |
| `out_of_scope` | Any query outside supported categories | — |

### Fine-Tuning Details

```python
from transformers import DistilBertForSequenceClassification, Trainer, TrainingArguments

model = DistilBertForSequenceClassification.from_pretrained(
    'distilbert-base-uncased',
    num_labels=len(INTENT_LABELS)
)

training_args = TrainingArguments(
    output_dir='./intent_model',
    num_train_epochs=5,
    per_device_train_batch_size=32,
    evaluation_strategy='epoch',
    save_strategy='best',
    load_best_model_at_end=True,
    metric_for_best_model='f1'
)
```

**Training data:** ~4,000 labeled query examples per intent class, augmented with paraphrase generation.

**Evaluation:** Macro F1 = 0.91 on held-out test set.

### Inference

```python
from transformers import pipeline

classifier = pipeline(
    'text-classification',
    model='./intent_model',
    tokenizer='distilbert-base-uncased',
    return_all_scores=True
)

result = classifier("What is my attendance percentage this month?")
# → [{'label': 'attendance_query', 'score': 0.94}, ...]

intent = max(result[0], key=lambda x: x['score'])
# If intent['score'] < 0.6 → fallback to RoBERTa or out_of_scope
```

## Component 2 — Named Entity Recognition (NER)

### Model

**Framework:** spaCy (`en_core_web_sm` as base, with custom entity types)

### Custom Entity Types

| Entity Type | Description | Examples |
|---|---|---|
| `DATE` | Absolute or relative date references | "yesterday", "last week", "March", "2026-04-01" |
| `EMPLOYEE_NAME` | Employee names | "Sarah", "Raj Kumar", "John" |
| `METRIC` | Attendance or performance metric names | "attendance rate", "work hours", "performance score" |
| `DEPARTMENT` | Department names | "Engineering", "HR", "Finance" |

### Custom NER Training

```python
import spacy
from spacy.training import Example

nlp = spacy.load('en_core_web_sm')
ner = nlp.get_pipe('ner')

# Add custom labels
for label in ['EMPLOYEE_NAME', 'METRIC', 'DEPARTMENT']:
    ner.add_label(label)

# Training loop
with nlp.select_pipes(enable=['ner']):
    for epoch in range(30):
        losses = {}
        for text, annotations in training_data:
            example = Example.from_dict(nlp.make_doc(text), annotations)
            nlp.update([example], losses=losses, drop=0.3)
```

### Date Normalization

Relative date expressions are resolved to absolute date ranges:

```python
def resolve_date(entity_text: str, reference_date: date) -> tuple[date, date]:
    if entity_text == "yesterday":
        d = reference_date - timedelta(days=1)
        return d, d
    elif entity_text == "last week":
        start = reference_date - timedelta(days=reference_date.weekday() + 7)
        return start, start + timedelta(days=6)
    elif entity_text == "this month":
        return reference_date.replace(day=1), reference_date
    # ... additional patterns
```

## Component 3 — Data Retrieval

Based on the detected intent and extracted entities, the chatbot constructs a data retrieval query against PostgreSQL or Redis.

### Intent-to-Query Mapping

```python
QUERY_MAP = {
    'attendance_query': """
        SELECT date, status, login_time, logout_time, work_hours
        FROM attendance_logs
        WHERE employee_id = :employee_id
          AND DATE(login_time) BETWEEN :from_date AND :to_date
        ORDER BY login_time DESC
    """,

    'performance_query': """
        SELECT score_date, score, classification,
               attendance_rate, avg_work_hours, late_login_count,
               idle_ratio, shap_summary
        FROM performance_scores
        WHERE employee_id = :employee_id
        ORDER BY score_date DESC
        LIMIT 30
    """,

    'team_attendance_query': """
        SELECT e.name, a.status, a.login_time, a.logout_time
        FROM attendance_logs a
        JOIN employees e ON e.id = a.employee_id
        WHERE e.manager_id = :manager_id
          AND DATE(a.login_time) = :query_date
        ORDER BY e.name
    """
    # ... other intent queries
}
```

Role enforcement is applied at the retrieval layer — employee accounts cannot execute manager-scoped queries regardless of query phrasing.

## Component 4 — RAG Pipeline

The RAG pipeline retrieves relevant context chunks before LLM generation. This grounds the LLM's response in actual system data rather than hallucinated answers.

### Architecture

```python
from langchain.chains import RetrievalQA
from langchain.vectorstores import FAISS
from langchain.embeddings import HuggingFaceEmbeddings

# Vector store (built from attendance policy docs, FAQ entries, help text)
embeddings = HuggingFaceEmbeddings(model_name='sentence-transformers/all-MiniLM-L6-v2')
vectorstore = FAISS.load_local('gs://attendance-ml-artifacts/faiss-index/', embeddings)

retriever = vectorstore.as_retriever(search_kwargs={"k": 4})  # top-4 chunks
```

### What Is Indexed in the Vector Store

| Content Type | Description |
|---|---|
| HR Policy Snippets | Attendance rules, leave policies, overtime policy |
| FAQ Entries | Common employee/manager questions and answers |
| Feature Descriptions | Plain-language descriptions of metrics and score components |
| Error Explanations | Common system error descriptions and resolutions |

The vector store is rebuilt whenever knowledge base content is updated. Rebuild script: `ml/chatbot/rebuild_faiss_index.py`.

### Pinecone (Alternative to FAISS)

For production deployments requiring managed vector storage and auto-scaling:

```python
import pinecone
from langchain.vectorstores import Pinecone

pinecone.init(api_key=PINECONE_API_KEY, environment='gcp-starter')
vectorstore = Pinecone.from_existing_index('attendance-kb', embeddings)
```

Toggle between FAISS (local/dev) and Pinecone (production) via `ml/config/chatbot_config.yaml`: `vector_store: faiss | pinecone`

## Component 5 — LLM Response Generation

### Model Options

| Model | Deployment | Context Window | Notes |
|---|---|---|---|
| **LLaMA 3 8B** | GPU Compute Engine (primary) | 8,192 tokens | Default production model |
| **Mistral 7B** | GPU Compute Engine (fallback) | 8,192 tokens | Used if LLaMA 3 inference endpoint is unavailable |

Both models are served via `vLLM` for high-throughput inference:

```bash
# LLaMA 3 inference server startup
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Meta-Llama-3-8B-Instruct \
  --port 8080 \
  --tensor-parallel-size 1 \
  --max-model-len 8192
```

### Prompt Template

```python
SYSTEM_PROMPT = """You are an AI assistant for the Attendance Tracking system.
Answer the user's question using only the data and context provided below.
Do not invent data. If the answer is not in the provided data, say so clearly.
Keep responses concise and factual. Format tabular data as markdown tables.

User role: {role}
Current date: {current_date}
"""

USER_PROMPT = """
Retrieved attendance data:
{retrieved_data}

Retrieved context:
{rag_context}

User question: {user_query}
"""
```

### LangChain Integration

```python
from langchain.llms import VLLMOpenAI
from langchain.chains import LLMChain
from langchain.prompts import ChatPromptTemplate

llm = VLLMOpenAI(
    openai_api_base="http://llm-inference-vm:8080/v1",
    model_name="meta-llama/Meta-Llama-3-8B-Instruct",
    max_tokens=512,
    temperature=0.1       # low temperature for factual, deterministic responses
)

chain = LLMChain(llm=llm, prompt=ChatPromptTemplate.from_template(USER_PROMPT))
response = chain.run(retrieved_data=data, rag_context=context, user_query=query)
```

## Fallback and Error Handling

| Condition | Behavior |
|---|---|
| Intent confidence < 0.6 (DistilBERT) | Retry with RoBERTa |
| Intent confidence < 0.6 (RoBERTa) | Route to `out_of_scope` response |
| Data retrieval returns empty result | Respond: "No records found for the specified period." |
| LLM inference timeout (> 8s) | Return cached response if available; otherwise: "Unable to generate a response right now. Please try again." |
| LLM hallucination guard | Response is validated against retrieved data; if response contains figures not in the data, it is replaced with a direct structured data response |

### Hallucination Guard (Post-Processing)

```python
def validate_response(response: str, retrieved_data: dict) -> str:
    # Extract numeric claims from response
    claimed_numbers = extract_numbers(response)
    data_numbers = extract_numbers_from_data(retrieved_data)

    for num in claimed_numbers:
        if not any(abs(num - d) < 0.5 for d in data_numbers):
            # Response contains a number not in the source data
            return format_structured_response(retrieved_data)  # fallback to raw data

    return response
```

## Session Management

The chatbot does not persist conversation history between user sessions. Each query is processed independently.

Within a session (browser tab open), a short-term context window of the last 3 turns is maintained to support follow-up questions:

```python
session_context = [
    {"role": "user", "content": "What is my attendance rate this month?"},
    {"role": "assistant", "content": "Your attendance rate this month is 92%."},
    {"role": "user", "content": "What about last month?"}
    # "last month" is resolved using the prior context
]
```
