# Testing Overview

This document describes the full testing strategy for the Attendance Tracking system — what is tested, which tools are used at each layer, test environment setup, how to run the full suite, and which tests block deployment.

---

## Testing Strategy

The system uses a layered testing approach. Tests are written closest to the code they validate and become broader as layers are added.

```
┌──────────────────────────────────────────────────────┐
│                  E2E / UI Tests                      │  ← Selenium, Playwright
│  Simulate real user flows through the full stack     │
├──────────────────────────────────────────────────────┤
│              Integration Tests                       │  ← pytest + test DB
│  Test blueprint routes with real DB and Redis        │
├──────────────────────────────────────────────────────┤
│                 API Tests                            │  ← Postman, Rest Assured
│  Validate endpoint contracts, auth, schemas          │
├──────────────────────────────────────────────────────┤
│               ML Model Tests                         │  ← pytest + fixtures
│  Validate inputs, outputs, accuracy, consistency     │
├──────────────────────────────────────────────────────┤
│                 Unit Tests                           │  ← pytest
│  Test individual functions and classes in isolation  │
└──────────────────────────────────────────────────────┘
```

---

## Test Types Matrix

| Type | Tool(s) | Scope | Trigger | Blocks Deploy? |
|---|---|---|---|---|
| Unit | pytest | Individual functions, helpers, utilities | Every push | Yes |
| Integration | pytest + TestContainers | Blueprint routes, DB interactions | Every push | Yes |
| API | Postman (Newman), Rest Assured | All REST endpoints | Every push | Yes |
| UI | Selenium, Playwright | Critical user flows (check-in, login, report export) | Pre-release | Yes |
| ML Model | pytest + fixtures | Model input/output, accuracy, consistency | Every push | Yes |
| Regression | Full suite re-run | Triggered on model update, feature deploy, config change | On trigger event | Yes |
| Performance / Load | Locust | API throughput under concurrent users | Pre-release | No (warns) |

## Test Environment Setup

### Local (Unit + Integration)

```bash
# 1. Create virtual environment
python -m venv venv && source venv/bin/activate

# 2. Install dependencies
pip install -r backend/requirements.txt -r backend/requirements-dev.txt

# 3. Start test dependencies via Docker Compose
docker compose -f docker-compose.test.yml up -d
# Starts: PostgreSQL (port 5433), Redis (port 6380)

# 4. Apply test database migrations
export DATABASE_URL="postgresql+psycopg2://test:test@localhost:5433/attendance_test"
alembic upgrade head
```

### `docker-compose.test.yml`

```yaml
services:
  postgres-test:
    image: postgres:14
    environment:
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
      POSTGRES_DB: attendance_test
    ports:
      - "5433:5432"

  redis-test:
    image: redis:7
    ports:
      - "6380:6379"
```

### CI Environment

The CI pipeline starts the same Docker Compose stack using GitHub Actions services:

```yaml
services:
  postgres:
    image: postgres:14
    env:
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
      POSTGRES_DB: attendance_test
    ports:
      - 5432:5432
```

## Running the Test Suite

### All Tests

```bash
# Run full suite
pytest backend/tests/ ml/tests/ -v --tb=short

# With coverage report
pytest backend/tests/ ml/tests/ -v --cov=backend/app --cov=ml --cov-report=html
open htmlcov/index.html
```

### By Type

```bash
# Unit tests only
pytest backend/tests/unit/ -v

# Integration tests only
pytest backend/tests/integration/ -v

# ML model tests only
pytest ml/tests/ -v

# E2E tests (requires running app server)
APP_URL=http://localhost:8080 pytest backend/tests/e2e/ -v
```

### API Tests (Postman via Newman)

```bash
# Install Newman
npm install -g newman

# Run collection
newman run api-tests/postman/AttendanceTracker.postman_collection.json \
  -e api-tests/postman/environments/local.json \
  --reporters cli,junit \
  --reporter-junit-export results/postman-results.xml
```

## Coverage Targets

| Module | Minimum Coverage |
|---|---|
| `app/blueprints/` | 85% |
| `app/services/` | 90% |
| `app/utils/` | 80% |
| `ml/models/` | 75% |
| `pipeline/` | 70% |

Coverage below minimum triggers a Slack warning but does not block deployment (thresholds are aspirational, not gates).

## CI Test Gate

The following checks must pass before the Docker build stage proceeds:

1. `flake8` linting — zero errors
2. `black` formatting check — no diff
3. All unit tests pass
4. All integration tests pass
5. All ML model tests pass
6. API tests pass (Postman collection)

Failure in any of these blocks the pipeline. E2E and load tests run in a separate pre-release workflow and do not block PRs.

## Regression Test Triggers

The full regression suite is re-run automatically when:

| Trigger | Action |
|---|---|
| ML model retrained and promoted | Run `ml/tests/` + integration tests |
| New feature merged to `main` | Run full test suite |
| Attendance rule configuration changed | Run `tests/integration/test_attendance_bp.py` |
| Geofence zone updated | Run GIS validation unit tests |
| Database migration applied | Run integration tests against migrated schema |

Regression runs are scheduled via GitHub Actions `workflow_dispatch` trigger and can also be initiated manually:

```bash
gh workflow run regression.yml --field reason="post-model-retrain"
```
