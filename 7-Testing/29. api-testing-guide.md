# API Testing Guide

This document covers the Postman collection structure, environment variable setup, JWT auth configuration, test case conventions, Rest Assured integration, and how to add tests for new endpoints.

---

## Postman Collection

**File:** `api-tests/postman/AttendanceTracker.postman_collection.json`

### Collection Structure

```

├── Auth (session_bp)
│   ├── Login — valid credentials
│   ├── Login — invalid password
```

## Environment Files

Environments are stored in `api-tests/postman/environments/`.

| File | Target |
|---|---|
| `local.json` | `http://localhost:8080` |
| `staging.json` | `https://api-staging.attendancetracker.yourdomain.com` |
| `production.json` | `https://api.attendancetracker.yourdomain.com` |

### Environment Variables

| Variable | Description |
|---|---|
| `base_url` | API base URL |
| `access_token` | Set automatically by the Login test |
| `manager_token` | Set by a separate manager login test |
| `admin_token` | Set by a separate admin login test |
| `employee_id` | Test employee UUID |
| `manager_id` | Test manager UUID |
| `attendance_record_id` | Set by checkin test; used in patch/delete tests |

## JWT Auth Setup in Postman

Authentication tokens are obtained and stored automatically using a Pre-Request Script in the collection root:

```javascript
// Collection Pre-Request Script
const tokenExpiry = pm.collectionVariables.get('token_expiry');
const now = Date.now();

if (!tokenExpiry || now > parseInt(tokenExpiry)) {
    pm.sendRequest({
        url: pm.variables.get('base_url') + '/auth/login',
        method: 'POST',
        header: { 'Content-Type': 'application/json' },
        body: {
            mode: 'raw',
            raw: JSON.stringify({
                email: pm.variables.get('test_employee_email'),
                password: pm.variables.get('test_employee_password')
            })
        }
    }, function (err, res) {
        const token = res.json().access_token;
        pm.collectionVariables.set('access_token', token);
        pm.collectionVariables.set('token_expiry', now + (8 * 3600 * 1000));
    });
}
```

All requests use `{{access_token}}` in the `Authorization: Bearer {{access_token}}` header via a collection-level Authorization setting.

## Test Case Structure

Each request includes a **Tests** tab with assertions. Standard assertion pattern:

```javascript
// Status code
pm.test("Status is 200", () => {
    pm.response.to.have.status(200);
});

// Response time
pm.test("Response time < 1000ms", () => {
    pm.expect(pm.response.responseTime).to.be.below(1000);
});

// Schema validation
const schema = {
    type: "object",
    required: ["record_id", "employee_id", "login_time", "status"],
    properties: {
        record_id:   { type: "string" },
        employee_id: { type: "string" },
        login_time:  { type: "string", format: "date-time" },
        status:      { type: "string", enum: ["present", "late", "half_day", "absent", "incomplete"] }
    }
};

pm.test("Response schema is valid", () => {
    pm.response.to.have.jsonSchema(schema);
});

// Business logic
pm.test("Status is 'present' for valid check-in", () => {
    const body = pm.response.json();
    pm.expect(body.status).to.eql("present");
});

// Capture for downstream tests
pm.collectionVariables.set("attendance_record_id", pm.response.json().record_id);
```

## Common Failure Patterns

| Failure | What It Indicates |
|---|---|
| `401 Unauthorized` | Token expired, not injected, or malformed `Authorization` header |
| `403 Forbidden` | Correct token but insufficient role for the endpoint |
| `422 Unprocessable Entity` | Request body failed WTForms validation — check `details` field |
| `404 Not Found` | Record ID is wrong or was not created by a prior test |
| Schema validation failure | API response shape changed — update schema in test and check backend for unplanned change |
| Response time > 1000ms | Investigate DB query performance or Cloud Run cold start |

## Running Postman Tests via Newman (CLI)

```bash
# Install Newman and HTML reporter
npm install -g newman newman-reporter-htmlextra

# Run against local environment
newman run api-tests/postman/AttendanceTracker.postman_collection.json \
  -e api-tests/postman/environments/local.json \
  --reporters cli,htmlextra \

  -e api-tests/postman/environments/staging.json \
  --bail   # stop on first failure
```

## Rest Assured (Java)

**Location:** `api-tests/rest-assured/`

Rest Assured tests are the secondary API test layer, used for contract regression testing in the CI pipeline.

### Setup

```xml
<!-- pom.xml -->
<dependency>
    <groupId>io.rest-assured</groupId>
    <artifactId>rest-assured</artifactId>
    <version>5.4.0</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.0</version>
    <scope>test</scope>
</dependency>
```

### Base Config

```java
// BaseApiTest.java
public class BaseApiTest {
    protected static String BASE_URL = System.getenv("API_BASE_URL");
    protected static String accessToken;

    @BeforeAll
    static void authenticate() {
        accessToken = given()
            .contentType("application/json")
            .body(Map.of("email", TEST_EMAIL, "password", TEST_PASSWORD))
        .when()
            .post(BASE_URL + "/auth/login")
        .then()
            .statusCode(200)
            .extract().path("access_token");
    }

    protected RequestSpecification authRequest() {
        return given()
            .contentType("application/json")
            .header("Authorization", "Bearer " + accessToken);
    }
}
```

### Example Test Cases

```java
// AttendanceApiTest.java
class AttendanceApiTest extends BaseApiTest {

    @Test
    void getMyAttendance_returns200WithRecords() {
        authRequest()
        .when()
            .get(BASE_URL + "/attendance/me")
        .then()
            .statusCode(200)
            .body("total", greaterThanOrEqualTo(0))
            .body("records", notNullValue())
            .body("records[0].record_id", notNullValue())
            .body("records[0].status",
                anyOf(is("present"), is("late"), is("half_day"), is("absent")));
    }

    @Test
    void getTeamAttendance_asEmployee_returns403() {
        authRequest()
        .when()
            .get(BASE_URL + "/attendance/team")
        .then()
            .statusCode(403)
            .body("error", equalTo("FORBIDDEN"));
    }

    @Test
    void checkin_missingFaceToken_returns422() {
        authRequest()
            .body(Map.of("latitude", 12.9716, "longitude", 77.5946))
        .when()
            .post(BASE_URL + "/attendance/checkin")
        .then()
            .statusCode(422)
            .body("error", equalTo("VALIDATION_ERROR"))
            .body("details.face_token", notNullValue());
    }
}
```

### Running Rest Assured Tests

```bash
cd api-tests/rest-assured/

# Run all tests
mvn test -DAPI_BASE_URL=http://localhost:8080

# Run specific class
mvn test -Dtest=AttendanceApiTest -DAPI_BASE_URL=http://localhost:8080

# Generate Surefire report
mvn surefire-report:report
open target/site/surefire-report.html
```

## Adding Tests for a New Endpoint

1. **Postman:** Add a new request to the relevant folder. Include:
   - Happy path (200/201)
   - Missing required fields (422)
   - Wrong role (403)
   - Not found (404) if the endpoint takes an ID
   - Capture response IDs in collection variables if downstream tests need them.

2. **Rest Assured:** Add a test method to the relevant `*ApiTest.java` class. Follow the `given().when().then()` pattern. Add negative cases (403, 422) alongside the happy path.

3. **Update the environment variables** if the new endpoint introduces new resource IDs that tests depend on.

4. Export the updated Postman collection (File → Export → Collection v2.1) and commit the JSON file.
