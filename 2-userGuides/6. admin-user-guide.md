# Admin User Guide

This guide covers all administrator functions — user and role management, geofence configuration, attendance rule setup, ML model scheduling, system health monitoring, and audit log access.

---

## Admin Panel Overview

Admins access the **Admin Panel** from the top navigation bar → **Admin Console**. The panel is only visible to accounts with the Admin role.

| Panel Section | Purpose |
|---|---|
| **User Management** | Create, edit, deactivate user accounts and assign roles |
| **Geofence Configuration** | Define and manage approved office zones |
| **Attendance Rules** | Set work hour expectations, late login thresholds, and half-day rules |
| **ML Scheduling** | Configure and trigger model retraining jobs |
| **System Health** | View platform status, error rates, and active sessions |
| **Audit Logs** | Access a full log of system and user actions |

---

## Managing User Accounts

### Creating a User
1. Go to **Admin Console → Users → Add User**.
2. Fill in: Full Name, Email, Mobile number, Department, Role (Employee / Manager / Admin).
3. Click **Create**. The system sends a welcome email with a temporary password and enrollment instructions.

### Editing a User
1. Search for the user by name or email in the Users table.
2. Click **Edit** on the user row.
3. Update fields as needed (name, department, role, manager assignment).
4. Click **Save Changes**.

### Deactivating a User
1. Open the user's profile.
2. Click **Deactivate Account**.
3. Confirm the action. The account is immediately locked; historical data is retained.

> Deactivated accounts cannot log in but their attendance and performance records remain accessible in reports and audit logs.

## Setting Attendance Rules

1. Go to **Admin Console → Attendance Rules**.
2. Configure the following parameters:

| Rule | Description | Default |
|---|---|---|
| **Work Start Time** | Expected login time | 09:00 |
| **Late Login Threshold** | Minutes after Work Start Time before a login is marked Late | 15 min |
| **Minimum Work Hours (Full Day)** | Hours required for a Present (full day) status | 8 hours |
| **Minimum Work Hours (Half Day)** | Hours required for a Half-Day status | 4 hours |
| **Idle Time Threshold** | Minutes of inactivity before idle time is logged | 30 min |
| **Working Days** | Which days of the week are counted as working days | Mon–Fri |

3. Click **Save Rules**. Rules apply globally unless overridden at the department level.

> Department-level overrides can be set by selecting a department from the dropdown at the top of the Attendance Rules page before saving.

## Managing ML Model Retraining

The system's ML models are retrained periodically to reflect updated attendance data. Admins can view the schedule and trigger manual retraining.

### Viewing the Retraining Schedule
Go to **Admin Console → ML Models**. The table shows each model, last trained date, next scheduled date, and status.

| Model | Default Retraining Frequency |
|---|---|
| Performance Prediction (XGBoost) | Monthly |
| Employee Classification (Logistic Regression) | Monthly |
| Anomaly Detection (Isolation Forest) | Bi-weekly |
| Trend Forecasting (Time-Series) | Weekly |

### Triggering a Manual Retraining Job
1. Click **Retrain** next to the relevant model.
2. Confirm the action. A retraining job is queued on Cloud Run Jobs.
3. Status updates to **Training** and then **Completed** once done (usually within 15–30 minutes depending on data volume).

> Manual retraining should be triggered after significant data changes — such as a large batch of employee additions or corrections to historical records.

## System Health Monitoring

1. Go to **Admin Console → System Health**.
2. Key indicators displayed:

| Metric | Description |
|---|---|
| **API Status** | Current availability of all backend endpoints |
| **Active Sessions** | Number of employees currently logged in |
| **Error Rate (1h)** | Percentage of API requests returning errors in the past hour |
| **ML Inference Latency** | Average response time for ML model queries |
| **DB Connection Pool** | Current usage of database connections |

3. Click **View Full Metrics** to open the Grafana monitoring dashboard (opens in a new tab; requires Grafana access).

## Accessing Audit Logs

Audit logs record all significant system and user actions.

1. Go to **Admin Console → Audit Logs**.
2. Filter by: Date Range, User, Action Type, Module.
3. Action types logged include:

| Action Type | Examples |
|---|---|
| Authentication | Login, logout, failed login attempts, password resets |
| User Management | Account creation, role changes, deactivation |
| Attendance | Manual attendance corrections, status overrides |
| Configuration | rule updates, ML schedule changes |
| Anomaly | Alert acknowledgement, escalation, false positive dismissal |

4. Export logs as CSV for compliance reporting.

> Audit logs are retained for **12 months** by default. Contact your DevOps team to adjust the retention period.
